Face recognition approach — your options
The most practical choice for Python is the deepface library. It wraps several state-of-the-art models (FaceNet, ArcFace, VGG-Face) under a single API, handles both detection and embedding, and works great with Flask/FastAPI. For detection specifically, RetinaFace and MTCNN are the top picks — RetinaFace is more accurate, MTCNN is lighter. For video, you'd extract frames using OpenCV, then run each frame through the same pipeline.
________________________________________
Database — your real decision
This is the most impactful architectural choice. You need two things stored: the face embeddings (numerical vectors ~128–512 dimensions) and the person's details (name, ID, etc.). Your options:
Option	Best for	Trade-offs
PostgreSQL + pgvector	Most projects	SQL + vector search in one DB, easy to self-host
FAISS (Facebook)	Speed at scale	In-memory, not persistent on its own
Pinecone / Weaviate	Cloud-native	Managed, easy to scale, paid
SQLite + numpy	Prototyping	Simple but slow at scale
For a starting project, PostgreSQL + pgvector is the sweet spot — you get relational storage for person details and fast vector similarity search in the same database.

Recommended tech stack to start with
•	FastAPI — for the REST API
•	deepface — for detection + embedding

FastAPI is your backend framework — the entry point for everything. When a user uploads a photo or video, FastAPI receives that file, routes it to the right processing logic, and returns the result as JSON. It's chosen over Flask here because it natively supports async operations (important when processing large videos), has built-in file upload handling, and auto-generates API documentation. Think of it as the "manager" that coordinates all the other tools.
________________________________________
Now here's the core of your system — the ML pipeline. This is where the real work happens, in two stages: detection, then embedding.
OpenCV is your media workhorse. For images, it decodes the file into a pixel array and resizes/normalises it so the ML models receive consistent input. For videos, it opens the file, seeks through it frame by frame (you'd typically sample every 5th or 10th frame rather than every single one for speed), and feeds each frame into the pipeline. Without OpenCV, you'd have no way to turn a .mp4 file into something a face detector can process.
RetinaFace / MTCNN are face detectors. Their job is not to recognise anyone — just to find where the face is in the image and draw a bounding box around it. They also align the face (rotate it so eyes are level) which dramatically improves recognition accuracy downstream. RetinaFace is more accurate and handles small/occluded faces better; MTCNN is lighter and faster.
DeepFace sits on top of detection and handles the recognition step. It takes the cropped face region and passes it through a deep neural network (you can choose FaceNet, ArcFace, or VGG-Face as the backend) which outputs a numerical vector — called an embedding — of 128 to 512 numbers that uniquely represents that face. Two photos of the same person will produce embeddings that are very close to each other in vector space; two different people will be far apart. This is the number you store in your database and compare against.
________________________________________
Now for the storage layer — the part you haven't decided yet.
PostgreSQL + pgvector gives you a single database that handles both things you need: regular relational data (person's name, ID, photo URL, any other profile fields) stored as normal SQL tables, and the face embedding vectors stored in a vector column that supports fast nearest-neighbour search. You query it like: "give me the person whose stored embedding is closest to this new embedding" — and if the distance is below your threshold, it's a match.
SQLAlchemy is the Python ORM that sits between FastAPI and PostgreSQL — it lets you write Python classes instead of raw SQL, handles database connections, and makes your code much cleaner.
________________________________________
In short, the data flow through your project looks like this:
Upload → FastAPI receives file
       → OpenCV decodes it (+ extracts frames if video)
       → RetinaFace detects & crops face(s)
       → DeepFace converts face to embedding vector
       → pgvector searches DB for nearest match
       → SQLAlchemy fetches person details
       → FastAPI returns JSON response
